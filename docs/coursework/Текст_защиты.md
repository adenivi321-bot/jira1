## Слайд 2. Актуальность темы (≈ 45 секунд)

Современные программные команды работают в распределённом режиме и не могут существовать без единой среды для постановки, декомпозиции и контроля задач. Зарубежные SaaS — Jira, Asana, Linear — закрывают эту потребность, но порождают свои проблемы: рост стоимости лицензий, привязку к вендору и жёсткие требования к локализации данных и платежам, особенно в корпоративной и государственной сфере.

Продуктовый ответ на это — **B2B‑SaaS с multi‑tenant изоляцией, тарифной моделью и интеграцией с локальной платёжной системой**, который при этом можно ставить и в облаке, и on‑premise. Современный технологический стек — NestJS, React, PostgreSQL — позволяет построить полноценный прототип такой системы силами одного разработчика. Именно этим обусловлена актуальность выбранной темы.

## Слайд 5. 1.1. Системы управления задачами (≈ 50 секунд)

Под системой управления задачами в работе понимается программное средство для регистрации, декомпозиции, планирования и контроля задач проектной команды.

К базовым функциям таких систем относятся:

- регистрация задач, ошибок и пользовательских историй;
- управление статусами и жизненным циклом задач;
- приоритизация и оценка трудозатрат;
- коммуникация — комментарии, уведомления, упоминания;
- отчётность и метрики: burndown, velocity, lead time.

Системы классифицируются **по модели поставки** — SaaS, on‑premise, self‑hosted; **по методологии** — Kanban, Scrum, Waterfall; **по арендности** — single‑tenant и multi‑tenant; **по монетизации** — free, freemium, subscription. Прототип TaskHub — это **multi‑tenant SaaS‑прототип с тарифной моделью** (FREE, BASIC, PRO, ENTERPRISE), разворачиваемый on‑premise через Docker, поддерживающий и Kanban, и Scrum.

## Слайд 6. 1.2. Сравнительный анализ аналогов (≈ 55 секунд)

Был проведён сравнительный анализ шести систем по четырём критериям: модель поставки, поддержка мультиарендности, тарифная модель и гибкость рабочих процессов.

Ключевые наблюдения:

- **Jira (Cloud + DC)** покрывает все функциональные потребности, но привязана к per‑user‑лицензированию и не даёт встроенной мультиарендности;
- **GitHub Projects, Asana, Trello** — облачные сервисы без режима on‑premise и без мультиарендности;
- **YouTrack** допускает on‑premise, но мультиарендность ограничена;
- **OpenProject** есть on‑premise и Community/Enterprise‑версии, однако арендаторов «из коробки» нет и процессы менее гибкие.

Незакрытая ниша — **on‑premise multi‑tenant решение с собственной тарифной моделью**, которое можно поставить как продукт российской команде. Именно в эту нишу позиционируется прототип TaskHub: он совмещает встроенную multi‑tenant изоляцию через RLS, четыре тарифных плана и интеграцию с YooKassa. Этот анализ, по сути, и сформулировал требования к практической части работы.

## Слайд 7. 1.3. UML 2.5 как инструмент проектирования (≈ 50 секунд)

В качестве языка моделирования выбран **UML 2.5** — стандарт OMG, поддерживающий 14 типов диаграмм, разделённых на структурные и поведенческие.

Структурные описывают **что** есть в системе: class, component, deployment, package. Поведенческие описывают, **как** система работает: use case, sequence, activity, state machine.

Для покрытия и структуры, и поведения прототипа из этого набора отобраны **одиннадцать диаграмм**: use case, class, ER, три sequence (логин, создание и перемещение задачи), activity, state, component, deployment и package. Этот комплект полностью документирует и статическую структуру, и динамику системы — что и было одной из задач курсовой работы.

## Слайд 8. 1.4. Технологический стек (≈ 45 секунд)

Технологический стек подобран по принципу зрелого open‑source с явной типизацией.

- **Frontend:** React 18, TypeScript, Vite — быстрый dev‑сервер и сборка; TanStack Query — серверное состояние; Zustand — клиентское; Tailwind CSS — оформление.
- **Backend:** NestJS 10 на TypeScript, TypeORM для работы с базой, class‑validator для DTO, Passport JWT для авторизации, Socket.IO для реального времени, BullMQ для отложенных задач.
- **Инфраструктура:** PostgreSQL 16 с поддержкой Row‑Level Security, Redis 7, MinIO для вложений, Docker Compose, обратный прокси Traefik, Prometheus для метрик.
- **Платежи:** интеграция с YooKassa для приёма платежей по тарифам — без жёсткой привязки приложения к иностранным платёжным провайдерам.

Стек последователен: один и тот же TypeScript на клиенте и сервере, общие типы через npm‑пакет.

## Слайд 11. 2.2. Архитектура (≈ 50 секунд)

Архитектура прототипа — классическая трёхуровневая, с асинхронной шиной событий.

- **Frontend** — single‑page application на React, общающийся с бэкендом по REST и WebSocket.
- **Backend** на NestJS состоит из **десяти модулей**: Auth, Tenants, Users, Projects, Issues, Search, Notifications, Health, Redis и **Billing**.
- **Billing** реализует продуктовую часть SaaS: `PlanLimitsService` проверяет лимиты тарифа (число проектов, пользователей, задач) с кешем в Redis, `PlanGuard` подключается декоратором к контроллерам, `PaymentsService` создаёт платежи в YooKassa и обновляет тариф арендатора после подтверждения.
- **Хранилища**: PostgreSQL для доменных данных с включённой Row‑Level Security; Redis — кеш, pub/sub‑шина и хранилище pending‑платежей; MinIO — объектное хранилище вложений.
- Реальное время реализовано через Socket.IO; модули внутри сервера развязаны через `EventEmitter`, что позволяет дописывать обработчики без модификации существующих сервисов.

Component‑диаграмма на слайде показывает все эти связи целиком.

## Слайд 12. 2.3. Доменная модель — UML Class (≈ 40 секунд)

Доменная модель формализована на диаграмме классов: **одиннадцать сущностей** — Tenant, User, Project, BoardColumn, Sprint, Issue, IssueComment, IssueAttachment, IssueChangelog, Label, Notification — и **пять перечислений** — UserRole, IssueStatus, IssuePriority, SprintStatus, BoardType.

Ключевая связь — композиция Tenant → Project → Issue: она моделирует владение данными арендатором. Эта диаграмма напрямую соответствует TypeORM‑сущностям в коде backend, что упрощает синхронизацию проекта и документации.

## Слайд 13. 2.4. Модель данных и многоарендность (≈ 50 секунд)

База данных состоит из **двенадцати таблиц** в нотации Crow’s Foot. Ключевая особенность — **обязательная привязка** всех доменных таблиц к таблице `tenants` через поле `tenant_id`, индексированное и обязательное.

Изоляция арендаторов реализована через **Row‑Level Security PostgreSQL**: middleware из JWT извлекает идентификатор арендатора, выставляет его в сессионную переменную, а RLS‑политика автоматически добавляет фильтр `tenant_id = current_setting(...)` к каждому запросу. Без корректного контекста запрос отклоняется на уровне СУБД — это серьёзная защита от случайной утечки данных между клиентами.

Дополнительно: первичные ключи UUID, ON DELETE CASCADE для целостности, полнотекстовый поиск по задачам через `tsvector` и GIN‑индекс, fractional indexing для drag‑and‑drop без массовых обновлений строк.

## Слайд 14. 2.5. Поведение системы (≈ 55 секунд)

Поведение проиллюстрировано двумя диаграммами.

**Sequence «создание задачи»** показывает синхронный путь — Web‑клиент, Controller, Service, Repository, PostgreSQL — внутри транзакции с одновременной записью в `issue_changelog`. Сразу после коммита публикуется событие `issue.created` через `EventEmitter`; на него асинхронно подписаны WebSocket‑шлюз и модуль Notifications. Это даёт реальное время для других пользователей и расширяемость без изменения сервиса задач.

**State machine** для сущности Issue фиксирует **шесть состояний** — BACKLOG, TODO, IN_PROGRESS, IN_REVIEW, DONE и CANCELLED — и допустимые переходы. Каждый переход — это конкретное API‑действие: `start`, `submitForReview`, `approve`, `reject` и так далее. Запрет недопустимых переходов реализован на уровне сервиса; все смены состояния попадают в журнал аудита.

## Возможные вопросы и краткие ответы

**1. Почему в self‑hosted‑прототипе есть подписка и тарифы — это же противоречит open‑source?**
TaskHub изначально проектировался не как community‑альтернатива Jira, а как **B2B‑SaaS‑прототип**. Поэтому полноценная продуктовая модель — multi‑tenant изоляция плюс тарифные планы (FREE / BASIC / PRO / ENTERPRISE) плюс приём платежей через YooKassa — заложена в архитектуру специально, чтобы продемонстрировать полный набор слоёв B2B‑SaaS, а не только трекер задач. Возможность поставки on‑premise через Docker и открытость кодовой базы — это **техническая** характеристика прототипа, а не бизнес‑модель: точно так же устроены, например, GitLab, Sentry, Mattermost — open‑core продукты с тарифами в той же кодовой базе.

**2. Как именно работает биллинг и что будет, если YooKassa не настроена?**
`PaymentsService` использует «verify‑by‑poll»‑подход: при создании платежа `paymentId` кладётся в Redis на 2 часа, после редиректа фронт дёргает `GET /billing/verify`, сервис тянет статус из YooKassa и при `succeeded` обновляет `tenant.plan`. Если переменные окружения `YUKASSA_SHOP_ID` и `YUKASSA_SECRET_KEY` не заданы — модуль логирует предупреждение и эндпоинты оплаты возвращают ошибку, но остальная система работает в режиме тарифа FREE с ограничениями по проектам, пользователям и задачам.

**3. Почему выбран NestJS, а не «голый» Express?**
NestJS даёт модульную структуру, DI и встроенную поддержку TypeScript «из коробки». Это ключевые требования для проекта с расчётом на дальнейшее расширение.

**4. Чем Row‑Level Security лучше фильтрации в коде приложения?**
RLS работает на уровне СУБД и срабатывает даже при ошибке программиста: без корректного `tenant_id` запрос вернёт ноль строк или ошибку. Это защита уровня БД, а не доверие приложению.

**5. Зачем нужен `EventEmitter`, если уже есть WebSocket?**
`EventEmitter` развязывает модули. Сервис задач не знает о существовании WebSocket‑шлюза или Notifications — он просто публикует событие. Подписчиков можно добавлять и убирать без изменений в core‑логике.

**6. Что такое fractional indexing и зачем он нужен?**
Это способ хранить порядок элементов как дробное число между соседями. Перетаскивание задачи требует обновить **одну** строку, а не пересчитывать индексы у целой колонки. Это ключ к комфортному drag‑and‑drop при больших досках.

**7. Какие диаграммы UML наиболее ценны для практики?**
В моём опыте — use case (фиксирует границы), class и ER (синхронизированы с кодом и БД), sequence (документирует нетривиальные сценарии вроде асинхронных событий) и deployment (нужен при передаче проекта в эксплуатацию).

**8. Сколько строк кода в прототипе и можно ли развернуть его одной командой?**
Да, развёртывание выполняется командой `docker compose up` — стек контейнеризован полностью: backend, frontend, PostgreSQL, Redis, MinIO и обратный прокси Traefik.

**9. Почему именно одиннадцать диаграмм, а не все четырнадцать из UML 2.5?**
Объект, состояние объектов, время и обзор взаимодействий (object, composite, timing, interaction overview) дублируют информацию, уже отражённую в выбранных диаграммах. Я отобрал минимально достаточный набор, покрывающий и структуру, и поведение.
